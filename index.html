01

¿Por qué importa el Naming?

La base de cualquier proyecto profesional en Unreal Engine

🔍

Búsqueda instantánea

Con prefijos estandarizados, puedes encontrar cualquier asset en segundos escribiendo solo el prefijo en el Content Browser. SM_ muestra todos los Static Meshes al instante.

👥

Trabajo en equipo

Un proyecto con naming consistente permite que cualquier miembro del equipo entienda la estructura inmediatamente, sin necesidad de explicaciones ni documentación adicional.

⚡

Evita errores costosos

Los assets sin nombre correcto generan referencias rotas, conflictos de merge en control de versiones y errores difíciles de rastrear en producción. El naming previene todo eso.

📈

Escalabilidad

Un proyecto bien nombrado desde el inicio puede crecer de 100 a 10,000 assets sin volverse un caos. La estructura correcta ahorra semanas de trabajo en proyectos grandes.

💡

Regla fundamental: Nunca uses espacios en nombres de assets o carpetas. Unreal Engine puede manejarlos, pero causan problemas en exports, scripts, control de versiones y pipelines de producción. Usa siempre PascalCase.

02

Tipos de Nomenclatura

Los formatos de escritura usados en Unreal Engine y cuándo aplicar cada uno

PascalCase
HeroCharacter

Cada palabra inicia con mayúscula. El estándar principal para nombres de assets, carpetas, clases y variables en UE5.

camelCase
heroCharacter

Primera palabra en minúscula, siguientes con mayúscula. Usado internamente en algunas variables de Blueprint, nunca en nombres de assets.

Snake_Case
Hero_Character

Palabras separadas por guion bajo. Usado en prefijos y sufijos de assets: SM_Hero_Sword_01. El separador de componentes del nombre.

SCREAMING_SNAKE
HERO_CHARACTER

Todo en mayúsculas con guion bajo. Reservado para macros y constantes en código C++. No usar en nombres de assets.

✅

En UE5: Usa PascalCase para todo. El guion bajo (_) se usa exclusivamente como separador entre el prefijo, el nombre base, variante y sufijo del asset.

03

El Patrón Universal de Naming

Una fórmula que aplica a todos los tipos de assets en Unreal Engine

Prefix_ Prefijo
+
BaseName Nombre base
+
_Variant Variante (opcional)
+
_Suffix Sufijo (opcional)
SM_Rock_Large_01 T_HeroArmor_D MI_GrassMaterial_Wet BP_EnemyBase SK_Character_Female AN_Run_Forward M_Metal_Brushed FX_Explosion_Big
💡

Nombre base compartido: Cuando un asset tiene múltiples archivos relacionados (mesh, textures, material), todos deben usar el mismo nombre base. SK_Hero, T_Hero_D, T_Hero_N, MI_Hero — el equipo sabe inmediatamente que pertenecen al mismo personaje.

Meshes & Geometría

Modelos 3D estáticos, skeletal meshes y geometría auxiliar

Prefijo Tipo de Asset Ejemplo Uso
SM_ Static Mesh SM_Rock_Large_01 Cualquier modelo 3D sin rig ni animación
SK_ Skeletal Mesh SK_HeroCharacter Modelos con esqueleto para animación
SKEL_ Skeleton SKEL_HeroCharacter El esqueleto/rig asociado al Skeletal Mesh
PM_ Physics Asset PM_HeroCharacter Configuración de física para ragdoll y colisiones
LB_ Level Bounds LB_ZoneSafeArea Límites de nivel para streaming
⚠️

Variantes numéricas: Cuando tengas múltiples versiones de un mismo asset, usa sufijos numéricos con ceros: SM_Rock_01, SM_Rock_02, SM_Rock_03. Esto asegura orden correcto al ordenar por nombre (01 antes que 10).

06

Materials

La familia de assets más grande — masters, instancias y funciones

Prefijo Tipo de Asset Ejemplo Uso
M_ Material (Master) M_Metal_Brushed Material base. Contiene toda la lógica. No se aplica directamente a meshes.
MI_ Material Instance MI_HeroArmor_Blue Instancia derivada del master. Es lo que se aplica al mesh. Modificable en runtime.
MF_ Material Function MF_CalculateFresnel Fragmento de lógica reutilizable entre materiales. Como funciones de código.
MPC_ Material Parameter Collection MPC_GlobalParams Parámetros globales compartidos entre múltiples materiales (día/noche, clima).
MISE_ Material Instance (SubSurface) MISE_Skin_Dark Instancias especializadas para subsurface scattering (piel, cera, hojas).
PP_ Post Process Material PP_OutlineEffect Materiales para efectos de post-proceso (outlines, vignette, etc.).
💡

Flujo correcto de materiales: Siempre crea un M_ Master primero con todos los parámetros expuestos. Luego crea tantas instancias MI_ como variantes necesites. Nunca apliques el Master directamente a un mesh en producción.

Textures

El asset más abundante del proyecto — sufijos por tipo de mapa

Prefijo / Sufijo Tipo Ejemplo Descripción
T_Name_D Diffuse / Albedo / Base Color T_HeroArmor_D Color base del material. El mapa más fundamental.
T_Name_N Normal Map T_HeroArmor_N Simula detalle de superficie sin geometría adicional.
T_Name_R Roughness T_HeroArmor_R Controla qué tan rugosa o pulida es la superficie.
T_Name_M Metallic T_HeroArmor_M Define qué partes de la superficie son metálicas.
T_Name_AO Ambient Occlusion T_HeroArmor_AO Sombras de contacto para añadir profundidad visual.
T_Name_E Emissive T_HeroSuit_E Mapa de emisión de luz propia del material.
T_Name_ORM ORM (Packed) T_HeroArmor_ORM Occlusion + Roughness + Metallic empacados en RGB. Optimiza memoria.
T_Name_H Height / Displacement T_RockCliff_H Mapa de desplazamiento para tessellation o parallax.
T_Name_OP Opacity / Alpha T_Leaves_OP Máscara de transparencia para materiales translúcidos.
T_Name_SSS Subsurface Scatter Color T_Skin_SSS Color de dispersión bajo la superficie (piel, cera).
HDR_ HDR / Environment HDR_StudioLight_01 Imágenes de entorno de alto rango dinámico para IBL.
T_Name_UI UI Texture T_ButtonPlay_UI Texturas exclusivas para interfaces de usuario.
✅

Texture packing: Siempre que sea posible, empaqueta Roughness + Metallic + AO en un solo mapa ORM. Esto reduce el número de muestras de textura en el shader y mejora la performance significativamente en escenas complejas.

Nunca coloques assets directamente en la raíz de Content/. Todo debe estar dentro de la carpeta del proyecto (Content/ProjectName/). Esto previene conflictos al migrar assets entre proyectos y mantiene el Content Browser limpio.

💡

Carpeta Developers: Cada miembro del equipo tiene su propia carpeta en Content/Developers/NombreDelArtista/ para pruebas personales. Nada de esta carpeta va a producción ni se commitea a control de versiones.

Ejemplo Guiado — Personaje Hero

Cómo organizar correctamente todos los assets de un personaje completo

Imagina que estás creando el personaje principal de un juego. Así se ve la estructura correcta de todos sus assets:

Content/ProjectName/Characters/Hero/ 
Hero/
│
├── Meshes/
│   ├── SK_Hero.uasset                ← Skeletal Mesh
│   ├── SKEL_Hero.uasset              ← Skeleton / Rig
│   └── PM_Hero.uasset                ← Physics Asset
│
├── Animations/
│   ├── ABP_Hero.uasset               ← Animation Blueprint
│   ├── BS2D_Hero_Locomotion.uasset   ← Blend Space 2D
│   ├── AN_Hero_Idle.uasset
│   ├── AN_Hero_Walk_Forward.uasset
│   ├── AN_Hero_Run_Forward.uasset
│   ├── AN_Hero_Jump_Start.uasset
│   ├── AN_Hero_Jump_Apex.uasset
│   ├── AN_Hero_Jump_Land.uasset
│   └── Montages/
│       ├── AM_Hero_Attack_01.uasset
│       └── AM_Hero_Death.uasset
│
├── Materials/
│   ├── MI_Hero_Body.uasset           ← Material Instance body
│   ├── MI_Hero_Armor.uasset          ← Material Instance armor
│   ├── MI_Hero_Face.uasset           ← Material Instance face
│   └── MI_Hero_Eyes.uasset
│
├── Textures/
│   ├── T_Hero_Body_D.uasset          ← Diffuse
│   ├── T_Hero_Body_N.uasset          ← Normal
│   ├── T_Hero_Body_ORM.uasset        ← Occlusion/Roughness/Metal
│   ├── T_Hero_Armor_D.uasset
│   ├── T_Hero_Armor_N.uasset
│   ├── T_Hero_Armor_ORM.uasset
│   ├── T_Hero_Face_D.uasset
│   ├── T_Hero_Face_N.uasset
│   └── T_Hero_Face_SSS.uasset        ← Subsurface Scatter
│
└── Blueprints/
    ├── BP_Hero.uasset                ← Blueprint principal
    └── BP_HeroController.uasset     ← Player Controller
✅

Nota el patrón: Todos los assets del personaje Hero comparten el mismo nombre base Hero. Cualquier miembro del equipo que vea T_Hero_Body_D sabe inmediatamente que es la textura Diffuse del body del personaje Hero. Sin ambigüedad, sin búsqueda innecesaria.

12

Reglas de Oro

Los principios fundamentales que nunca debes romper

🚫

Sin espacios en ningún nombre

Nunca: Hero Character. Siempre: HeroCharacter. Los espacios rompen scripts, exports y pipelines automatizados.

📁

Todo dentro de ProjectName/

Nunca pongas assets directamente en la raíz de Content. La carpeta del proyecto es el contenedor de todo tu trabajo.

🔤

Siempre con prefijo

Si no sabes el prefijo exacto, búscalo antes de guardar. Un asset sin prefijo es imposible de identificar en un proyecto grande.

🔗

Nombre base compartido

Todos los assets relacionados (mesh, textures, material, blueprint) deben compartir el mismo nombre base para facilitar búsquedas y relaciones.

🔢

Números con ceros

Usa _01, _02, no _1, _2. Esto garantiza orden correcto alfabéticamente y permite hasta 99 variantes.

⚠️

Nunca renombrar assets referenciados

En UE5 puedes renombrar y el engine actualiza las referencias, pero si compartes en equipo con control de versiones, coordinalo antes para evitar referencias rotas.

🧹

Carpeta Developers = área de pruebas

Lo que está en tu carpeta personal de Developers nunca va a producción. Es tu sandbox. Cuando algo esté listo, muévelo a la carpeta correcta del proyecto.

📐

Consistencia sobre preferencia

Si el equipo ya tiene un estilo establecido, síguelo aunque no sea tu favorito. La consistencia del proyecto siempre supera la preferencia individual.
